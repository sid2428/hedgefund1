# Deployment

## Why not all of it on Vercel

Vercel runs serverless functions with an execution-time limit. Mosaic's
ingestion path fetches filings from EDGAR under an 8 req/s budget and runs LLM
extraction over them — minutes of work per company, in a **Celery worker** that
must stay resident. Serverless has nowhere to put that. It also needs
**PostgreSQL with pgvector** and **Redis**, neither of which Vercel provides.

So the frontend goes on Vercel and everything else goes on a container host.
This is the same split `render.yaml` describes in the Grow repo.

| Component | Host | Why |
|---|---|---|
| React frontend | Vercel | Static build, CDN, preview deploys |
| FastAPI backend | Render / Railway / Fly | Long-lived process |
| Celery worker | Same host, separate service | Persistent, no HTTP timeout |
| PostgreSQL + pgvector | Neon or Render Postgres | pgvector extension required |
| Redis | Upstash or Render | Celery broker |

## 1. Database first

Provision Postgres, then enable the extension **before** migrating — `001`
creates the vector index and will fail without it:

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

Then, from the backend directory:

```bash
alembic upgrade head
```

This applies `001_initial_schema`, `002_financial_facts` and
`003_temporal_edges`. **Run it against a throwaway database first.** `002` and
`003` are hand-written rather than autogenerated, and `003` alters a unique
constraint and backfills `known_from` from `created_at`. Confirm both apply
cleanly before pointing them at anything you care about.

Verify:

```sql
\d financial_facts
\d company_relationships
SELECT conname FROM pg_constraint WHERE conname = 'uq_company_rel';
```

The constraint should include four columns, `known_from` among them.

## 2. Backend

Container build is `backend/Dockerfile`. Start command:

```bash
uvicorn app.main:app --host 0.0.0.0 --port $PORT
```

Environment:

```
DATABASE_URL=postgresql+asyncpg://USER:PASS@HOST/DB
REDIS_URL=rediss://...
GROQ_API_KEY=...
OPENAI_API_KEY=...
EDGAR_USER_AGENT=Your Name your@email.com     # SEC requires a real contact
ENVIRONMENT=production
ALLOWED_ORIGINS=https://your-app.vercel.app
```

Two that bite:

- `DATABASE_URL` **must** use the `postgresql+asyncpg://` driver. Managed
  providers hand you `postgres://`, which the config validator rejects.
- `EDGAR_USER_AGENT` must contain an email. SEC returns 403 to generic agents,
  and the validator enforces it at startup rather than letting you find out in
  production.

## 3. Celery worker

Same image, different command, no HTTP port:

```bash
celery -A app.tasks.celery_app worker --loglevel=info -Q filings,theses
```

Both queues are required — `filings` handles ingestion, `theses` the agent
pipeline. A worker listening to only one will silently accumulate a backlog on
the other.

---

## 4. Frontend on Vercel

Import the repo, set **Root Directory = `frontend`**. `vercel.json` handles the
Vite build and SPA fallback.

Edit the API rewrite in `frontend/vercel.json`, replacing
`REPLACE_WITH_BACKEND_HOST` with your backend hostname. Proxying through the
same origin avoids CORS entirely and keeps the backend URL out of client
JavaScript.

Then set `CORS_ORIGIN` / `ALLOWED_ORIGINS` on the backend to the Vercel domain.

## 5. Before calling it live

- [ ] `alembic upgrade head` applied cleanly on a throwaway DB **first**
- [ ] `GET /health` and `/ready` respond
- [ ] One `POST /api/companies/{ticker}/ingest` completes end to end
- [ ] Worker logs show `xbrl_facts_ingested` with a non-zero count
- [ ] Worker logs show `extractor_grounding` with an `ungrounded_rate`
- [ ] `GET /api/graph/as-of?as_of=` returns different graphs for different dates
- [ ] `.env` is not committed anywhere

## Known gaps

Stated plainly rather than discovered later:

1. **The Postgres upsert path is untested.** `bulk_upsert` branches on dialect;
   CI exercises SQLite. The `ON CONFLICT DO NOTHING` path has never run. Watch
   the first ingest for duplicate-key errors.
2. **The graph builder does not set `known_from`.** It falls back to
   `date.today()`, so edges are stamped with ingestion time rather than filing
   date. Historical graph queries are only as accurate as that gap until the
   builder passes the filing date through.
3. **The frontend predates the temporal API.** It calls `/api/graph`, not
   `/api/graph/as-of`, so the point-in-time capability is not yet visible in the
   UI.
4. **No API rate limiting** on Mosaic's own endpoints.
5. **Render free tier sleeps.** First request after idle takes ~50s. Fine for a
   demo, not for a live audience — warm it before recording.
